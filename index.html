PROJECTS

Multiclass Leaf Image Classifier

in collaboration with Colin Probasco

Summary

We trained a classifier to identify the species of a tree given a photo of one of its leaves. The data for the project was a subset of the LeafSnap photo database consisting of 120 photos for each of 11 different species of trees.

 

We used two sets of shape-based features: a measure of convexity and up to 20 harmonics of the Elliptic Fourier Descriptors (EFDs). These features were extracted from each image by first segmenting the images to allow access to a clearly-defined contour.

 

The classifier was implemented as a multi-class support vector machine in an all vs all approach.

Why is this cool?

There are cool potential applications like

      - an automated field guide (the team that collected the LeafSnap data created an iOS field guide)

      - agricultural robots might use computer vision systems trained on plant data to tag sick plants or autonomously weed fields.

 

We also encountered many interesting computer vision questions like

      -  what's the best way to systematically segment images?

      -  what are good features, and how do we extract them?

       - how do we deal with noisy image data?

       - how do we process large amounts of data quickly?

4 species of cherry, 3 species of maple,  4 species of oak

Segmentation

LeafSnap included pre-segmented binary images, but the segmentation was incorrect for many of the leaves we were interested in so we designed our own.

To be fair, their segmentation algorithm was designed for 185 different species, so it's understandable that it performed worse for some species. Ours performed worse than the LeafSnap algorithm on some other species, for instance, the top-hat transform masks thin shapes like stems, but this messes up on pine needles.

 

         raw data       | LeafSnap segmentation |   our segmentation

We segmented images by:

      1.  Convert to binary images (all pixels are either 0 or 1)

      2.  Black out the pixels corresponding to the color and light scales   (this is the problem in the LeafSnap segmentation above)

      3.  Apply 3 'top-hat' transformations to automatically mask out the stems

      4.  Make all pixels within the contour white   (this is important to compute convex hull correctly)

      5.  Apply a filter to remove any artifacts

Shape Features

There have been a lot of experiments for different features for leaves.

 

The two features we used were:

      1.  (Area(shape) - Area( Convex hull(shape))

                                     Area(shape)

This tells you how close to convex a shape is. A Sargent cherry leaf is very close to convex while a Japanese maple leaf is very not-convex.

 

      2.  The 4 coefficients of the first ten harmonics of the Elliptic Fourier Descriptors (EFDs) of the shape.

What a mouthful! For a detailed description of this algorithm please check out my blog post.

 

Briefly, an EFD is a discrete Fourier series of the contour, which can be described by 4 coefficients. The number of harmonics is the number of terms in the series.

 

The pictures above illustrate how well EFDs can fit various shapes- the lines in red are the first ten harmonics, while the lines seen faintly below in blue are the actual contours. How cool is that!

Results

Overall, our 11-class classifier had a 58% accuracy. While this isn't *great*, keep in mind that random-chance over 11 classes .09%.

The table shows the accuracies of each binary classifier. Some classifiers, like Japanese Maple vs. Bur Oak have very good accuracy (99%), while others, like Norway Maple vs. Yoshino Cherry have much worse accuracy (65%).

We could improve the multi-class accuracy by adding features that improved the accuracy of the binary classifiers that performed the worst.

What next?

There are a couple of interesting ways to continue this project:

 

      1.  Hierarchical classification

Hierarchical classification is a really interesting multi-label scheme; for example, a Yoshino Cherry would be labeled as both a Yoshino Cherry and more generally as a cherry tree. The fantastic application would be given a species of cherry tree that the classifier was not trained on, it should identify that the input was both a cherry tree and not one of the classes of cherry tree that it knew how to label.

 

      2.  Fine-tune

We could try more features- we'd get the most bounce for the ounce by designing features that differentiate the types of leaves our binary classifiers had the most trouble with.

We could also make our segmentation more robust across different species so we can use more species.

Most of a Python Compiler

or how I learned to stop worrying and love functional programming

Summary

Lexing

I took Dr. Matt Might's Compilers class Spring 2015, in which we undertook building a full Python 3.4 compiler.

We didn't finish- partly because Dr. Might was asked to join the initiative for Precision Medicine half-way through the semester. (How cool!)

 

I'll describe at a high-level the stages of the compilation pipeline that we did implement

                                        Lexing -> Parsing -> Desugaring & Normalizing

and talk about some of the interesting corner cases.

Lexical analysis is the first step in compilation. The lexer looks at each word or symbol in your program file, and outputs one of a few recognized tokens to identify what the role of that word or symbol is. It's sort of like finding the 'python part-of-speech' of each word.

 

The lexical tokens we used are:

  * IDENTIFIER (variable names)

  * LITERAL (like the value 2, or the string "fox")

  * KEYWORDS (like "for", "break", "if", etc)

  * PUNCTUATION (like "+", "*", etc)

  * INDENT

  * DEDENT

  * NEWLINE

  * ENDMARKER

This nice ouroborus is from a 15th century manuscript.

We lex a python file by walking through the characters in the file and matching sequences of characters to the above categories. For example, the output for a python file that goes "x = 5" is the list

                       (ID "x")

                       (PUNCT "=")

                       (LIT 5)

                       (NEWLINE)

                       (ENDMARKER)

 

Some of the interesting corner cases included dealing with non-ascii unicode characters, keeping track of indentation and logical newlines and dealing with identifiers that contained keywords, like a boolean flag called 'is_a_leaf'.

 

For a slightly longer discussion, check out this blog post.

Parsing

Parsing comes next in the compilation pipeline- the parser takes a stream of tokens (produced by the lexer) and outputs an abstract syntax tree. Our AST matched Python's specified documentation.

 

The AST is just a specification of how different structures are nested in python, like that in "a+b", the "+" is an operator which is nested under the binary expression "a <operator> b".

 

Lucky for me, Dr. Might provided a framework for us which parsed Python's formal grammer into a form that we can represent in racket. As an example, the formal grammer rule:

                        xor_expr: and_expr ('^' and_expr)*

becomes the racket-friendly expression:

                       (xor_expr (seq and_expr (rep (seq "^" and_expr))))

 

We processed this racket representation of the python grammar to using Dr. Might's tool derp, and then some more of Dr. Might's framework used Racket's cfg parsing tools to a output valid python AST.

 

I'm going to show my implementation of the same xor expression rule below, and it looks scary, but hear me out, I'll explain it below:

 

                        (xor_expr

              ($--> (seq

                        and_expr

                        (rep (seq ($--> "^" 'BitXor) and_expr)))

                    (process-binops ($ 1) ($ 2))

          ))

 

         Here's what all those things above mean:

 

The symbol $--> just specifies that this is a reduction rule. The rule is formatted like

         ($--> (what-should-I-match?)

                    (what-should-I-do-with-what-I-matched?))

 

'seq' means concatenation, so (seq a b) matches the list (a b) but not the list (a).

When we match a sequence, we can access each element by number- so if we match (seq a b c) then ($ 1) is bound to a, ($ 2) is bound to b, ($ 3) is bound to c.

'rep' means 0+ repetitions, (as in Kleene star), so (rep a) matches the following lists: ( ) (a) (a a) (a a a.. a)

($--> "^" 'BitXor) is a reduction that means replace the string "^" with the symbol 'BitXor.

Once we've made the match we call a method called process-binops with 2 arguments. ($ 1) is the first mandatory "and expression" and ($ 2) is a possibly empty list with more "and expressions".

 

I'll spare you the code for process-binops, but at the end I output a racket list that looked something like:

                                        ('BinOp <and expression>).

If we matched anything in the 'rep' the output looks something like:

                                        ('BinOp <and expression> 'BitXor

                        ('BinOp <and expression> 'BitXor

                                 ...

                                 ('BinOp <and expression>)).

 

This matches the AST specification that a binary operation is

                          BinOp(expr left, operator op, expr right)

because 'BitXor is an operator and <and expression>s are expressions.

 

For reference, the reduction above is 6 lines of an 800 line-ish implementation. Yikes! This was definitely the hardest part of the compilation pipeline.

Desugaring & Normalizing

After we have the AST, we want to make all the elements as simple as possible because we want to end up at assembly. Many of the transformations are putting expression into canonical form, or dropping notation related to function calls like decorators, defaults and annotations.

 

We make a handful of transformations, including, for example:

 

      1.       return                 -->   return None

 

      2.       (a, b, c) = somelist  -->   $tmp = list(somelist)

                                            a = ($tmp)[0]

                                            b = ($tmp)[1]

                                            c = ($tmp)[2]

 

      3.       for loops               --> while loops

 

      4.       lambda expression       --> normal old method

 

Some of the transformations required us to first normalize the AST. Normalizing means that everything on the right-hand side of an assignment statement is a simple expression. For example

 

          annie_happy = drink( heat( water ) + teabag )

 

would normalize as:

 

                         a = heat( water)

          b = a + teabag

          annie_happy = drink( b )

 

The most interesting transformations were eliminating classes and normalizing multiple comparisons so that a < b < c  looked more like (a < b) and (b < c).
